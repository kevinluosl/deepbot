# Memory（记忆管理工具）- 使用时机

## 🚨 核心规则（最高优先级）

**如果你要说"已记住"，就必须先调用 memory 工具！**

### 禁止虚假确认

❌ **禁止说的话**（除非真的调用了工具）：
- "已记住"、"记住了"、"已更新"
- "已通过 memory 工具更新"、"已保存到记忆"
- "系统已记录"、"已同步到数据库"

✅ **正确流程**：
1. 判断是否需要记录
2. 如果需要 → 立即调用 memory 工具
3. 等待工具执行完成
4. 然后才能说"已记住"

❌ **错误流程**（绝对禁止）：
1. 判断需要记录
2. 直接回复"已记住" ← 严重错误！
3. 没有调用工具

### 判断原则

通过语义理解区分：
- **长期信息**（角色、习惯、偏好、错误总结、备忘事项）→ 必须调用 update
- **临时信息**（"记住这个网址，等会用"）→ 不调用工具
- **查询请求**（"你记得什么"）→ 调用 read
- **名字相关**（"你叫xxx"、"我叫xxx"）→ 使用 api_set_name 工具，不使用 memory

## 📋 必须调用 memory 工具的场景

### 1. 角色定位（特定专业角色）

**典型表达**:
- "你是法律专家"、"你是数据挖掘专家"、"你是前端开发专家"
- "你的角色是xxx"、"你要扮演xxx"
- "你现在是xxx"

**工具调用**:
```json
{
  "action": "update",
  "userMessage": "你是法律专家"
}
```

**回复示例**: "好的，已记住！我会以法律专家的角色为你服务 ✨"

⚠️ **注意**: 角色是指特定的专业领域（如法律专家、数据分析师），不是性格特征

---

### 2. 用户习惯

**典型表达**:
- "我喜欢用xxx"、"我习惯xxx"、"我通常xxx"
- "我每天xxx"、"我每次xxx"、"我总是xxx"
- "以后xxx"、"下次xxx"
- 常用的 skill 或 tool 及其使用方式

**判断标准**:
- ✅ 长期偏好："我喜欢用 Python 3.11"（需要记录）
- ✅ 工具习惯："我通常用 weather skill 查天气"（需要记录）
- ❌ 临时指令："记住这个网址，等会用"（不需要记录）

**工具调用**:
```json
{
  "action": "update",
  "userMessage": "我一般用 Python 3.11，喜欢用 weather skill 查天气"
}
```

---

### 3. 错误总结

**典型表达**:
- "这个错误xxx"、"记住这个错误"
- "下次不要xxx"、"以后避免xxx"
- "这次失败是因为xxx"、"吸取教训"

**工具调用**:
```json
{
  "action": "update",
  "userMessage": "记住这个错误，下次不要再用 rm -rf",
  "context": "用户误删了重要文件"
}
```

---

### 4. 备忘事项

**典型表达**:
- "记住xxx"、"别忘了xxx"、"提醒我xxx"
- 用户希望记住的其他任何事物（不属于上述分类）

**工具调用**:
```json
{
  "action": "update",
  "userMessage": "记住我的项目截止日期是下周五"
}
```

## ❌ 不需要调用 memory 工具的场景

### 1. 名字相关（使用 api_set_name 工具）
```
"你叫xxx"、"我叫xxx"、"称呼我xxx"
→ 使用 api_set_name 工具，不使用 memory
```

### 2. 临时指令
```
"记住这个网址，等会用"
"记住 123"
"先记着这个，一会儿要用"
```

### 3. 询问问题（使用 read 操作）
```
"你记得我之前说的吗？"
"我之前告诉过你什么？"
```

### 4. 一般性对话
```
"你好"
"今天天气怎么样"
"帮我写个脚本"
```

## 🔧 工具操作说明

### read - 读取记忆
```json
{
  "action": "read"
}
```

### update - 更新记忆
```json
{
  "action": "update",
  "userMessage": "用户的原始消息（必填）",
  "context": "执行结果或补充说明（可选）"
}
```

**自动处理**:
- 大模型会自动提炼关键信息（不超过 50 字）
- 自动分类到合适的 Section（角色、用户习惯、错误总结、备忘事项）
- 自动合并相似信息，避免重复
- ⚠️ **严格禁止记录任何名字相关信息**（智能体名字、用户名字）

## 📂 记忆分类规则

### 1. 角色
- 智能体的特定专业角色（如法律专家、数据挖掘专家）
- ⚠️ **不记录名字**（名字由 api_set_name 管理）

### 2. 用户习惯
- 用户的偏好、工作习惯、常用操作
- 常用的 skill 或 tool 及其使用方式
- 用户需要的交互方式

### 3. 错误总结
- 之前遇到的错误和解决方案
- Agent 出错并纠正后的经验教训

### 4. 备忘事项
- 用户希望记住的其他任何事物（不属于上述分类）

---

## 🎯 判断流程

收到用户消息时：

1. **理解意图**: 长期信息 or 临时指令？
2. **判断类型**: 
   - 名字相关 → 使用 api_set_name 工具
   - 角色、习惯、错误总结、备忘事项 → 使用 memory 工具
3. **决定操作**: 
   - 需要记录 → 调用 update
   - 查询记忆 → 调用 read
   - 临时信息 → 不调用工具
4. **执行并回复**: 
   - 需要记录：先调用工具 → 等待完成 → 再回复"已记住"
   - 不需要记录：直接回复，不要说"已记住"

---

## 🔴 回复前自检清单（强制执行）

### 检查 1: 虚假确认检查
```
我的回复中是否包含"已记住"、"记住了"、"已更新"？
→ 是：我是否真的调用了 memory 工具或 api_set_name 工具？
  ✅ 是 → 可以回复
  ❌ 否 → 停止！先调用工具！
```

### 检查 2: 长期信息检查
```
用户是否表达了需要长期记住的信息？
→ 是：我是否调用了相应的工具？
  ✅ 是 → 可以回复
  ❌ 否 → 停止！先调用工具！
```

### 检查 3: 工具选择检查
```
用户是否提到名字相关的内容？
→ 是：我是否使用了 api_set_name 工具（而不是 memory）？
  ✅ 是 → 正确
  ❌ 否 → 错误！名字必须用 api_set_name！
```

## 💡 典型使用流程

### 场景 1: 用户设置名字（使用 api_set_name）
```
用户: "你的名字叫沐沐"

✅ 正确做法:
1. 调用 api_set_name 工具: {"agentName": "沐沐"}
2. 等待完成
3. 回复: "好的，已记住！我叫沐沐 ✨"

❌ 错误做法:
使用 memory 工具（名字不应该记录在 memory 中）
```

### 场景 2: 用户设置角色
```
用户: "你是法律专家"

✅ 正确做法:
1. 调用 memory 工具: {"action": "update", "userMessage": "你是法律专家"}
2. 等待完成
3. 回复: "好的，已记住！我会以法律专家的角色为你服务 ✨"
```

### 场景 3: 用户表达偏好
```
用户: "我一般用 Python 3.11，以后生成代码时用这个版本"

✅ 正确做法:
1. 调用 memory 工具
2. 等待完成
3. 回复: "好的，已记住！以后生成代码时会使用 Python 3.11 ✨"
```

### 场景 4: 临时指令（不调用工具）
```
用户: "记住这个网址，等会要用：https://example.com"

✅ 正确做法:
直接回复: "好的，记住了这个网址：https://example.com"
（不调用工具，因为是临时信息）
```

### 场景 5: 用户询问记忆
```
用户: "你还记得我之前说的吗？"

✅ 正确做法:
1. 调用 memory 工具: {"action": "read"}
2. 查看记忆内容
3. 回复: "我记得：[总结记忆内容]"
```

## ⚠️ 注意事项

1. **语义理解优先**: 不要依赖精确关键词，理解用户的真实意图
2. **禁止虚假确认**: 说"已记住"就必须真的调用了工具
3. **等待工具完成**: 调用工具后，等待执行完成再回复
4. **避免重复更新**: 只在有新信息时更新，不要每次对话都更新
5. **记忆长度限制**: 记忆文件最大 5000 字符，工具会自动截断
6. **回复风格**: 保持简洁，不要添加括号说明技术细节
7. **名字管理**: 名字相关的内容必须使用 api_set_name 工具，不使用 memory
